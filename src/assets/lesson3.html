If the view could only hold static data, there would be no need to learn Mithril. We can funnel data into the view from several sources. As we shall see, the view can print the data or alter it to enhance its presentation.

Data from the contoller

see src/app.coffee

In addition to the view function, a component often contains a controller function. Like the view, it returns an object.

Although there are intermediate steps that we'll cover later, program flow is generally like this:
m.mount(el, App) -> App.controller() -> App.view() -> browser
Note that if the controller is omitted, in the background Mithril supplies one equal to a function that returns an empty object. Among the controller's many responsibilities, it may pass data to the view. This happens automatically. The view receives the returned object from the controller as a parameter. By convention, we call it ctrl.

In the code, the controller supplies data to the view in the object key/value pair: displayLogo: true, a boolean switch that turns the Logo component on and off. The view accesses it in the passed-in object: ctrl.displayLogo, then displays the Logo depending on it's value.

Component agruments

It's possible to pass arguments to a component before it's rendered to the virtual DOM using the method: m.component. An "args" object is passed to both the controller and the view. A common technique is to provide default values within the component and optionally override those defaults in the arguments object.

// App.view
m.component(Header, {title: "Lesson 3"})
...
// Header
controller: function(args){... //args.title = "Lesson 3"
view: function(ctrl, args){... //args.title = "Lesson 3"

Data Model

In MVC, M is for model, V is for view, and C is for controller. Now let's look at the role the model plays in supplying data to the view.

The model acquires data from an outside source and passes it on to the rest of our app. Data could come from a REST-ful connection to a database, a websocket, the browser's localstorage, or any type of web service like Twitter or Google Maps. In this lesson we are going to hard code the data in the model object.

See src/components/Footer.coffee

The model is usually an object with data properties and possibly functions that massage the data. It can be internal to a component, but it's often defined outside to give data access to a controller, a view, and even other components.

In the next lesson we'll see MVC in action. We'll learn how to control the flow of data through our app: how to retrieve it from the server, display it on the screen, and receive input from user interaction.
